## Core operating rules

You must follow these rules throughout:

1. Treat AI as an accelerator, not as an authority.
2. No implementation should bypass deterministic risk controls.
3. Never allow trading logic, webhook handling, or AI output to directly place live trades without passing hard validation and risk gates.
4. Every service boundary, deployment choice, and infrastructure decision must be justified.
5. Prefer simple, robust, observable architecture over clever complexity.
6. Design for greenfield, but avoid lock-in to one exchange or one deployment platform.
7. The system must support multiple exchange APIs for live trading, with Binance definitely included, and at least one or more of: Bybit, OKX, Kraken, Coinbase, depending on what is most appropriate.
8. The system must also support TradingView integration safely through webhook ingestion or a similarly safe event path.
9. Hosting/deployment context:
   - local development and operator environment will use Podman
   - self-hosted deployment/orchestration will use Coolify where appropriate
10. Clearly separate:
   - what should run locally in Podman for development, testing, and paper trading
   - what should be deployed/managed in Coolify for persistent hosted operation
11. Do not over-engineer for HFT if the assumptions do not require it.
12. Do not assume one exchange’s API behavior applies to all exchanges.
13. Secrets, API keys, webhook tokens, and exchange credentials must never be hardcoded.
14. Every live-trading path must have audit logging, replayability, and operator visibility.
15. If any part of the current proposal is unsafe, fragile, or too complex for phase 1, say so clearly and propose a safer alternative.
